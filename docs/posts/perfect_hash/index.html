Perfect Minimal Hashing

Author: Matt Piekenbrock
Date: 2022-12-03

I am not an expert in hashing. However, I was confronted with the following problem the other day:

Suppose you have a fixed set of nnn distinct integers S={s1,s2,…,sn}S = \{s_1, s_2, \dots, s_n \}S={s1​,s2​,…,sn​}. Construct an efficient hash function h:S→[n]h: S \to [n]h:S→[n] that maps every element s∈Ss \in Ss∈S to an integer in the range [n]={1,2,…,n}[n] = \{1, 2, \dots, n\}[n]={1,2,…,n} in such a way that you are guaranteed O(1)O(1)O(1) lookup and O(n)O(n)O(n) storage.

Assume that SSS is fixed and that you have an infinite amount of preprocessing time.

Simple, right? I, admittedly, didn't know how to do it. It turns out what I wanted was a practical and perfect minimal hashing function.

In this post, I'll give an elegant way to solve this problem. Before doing so, here's an example problem whose solution is given by solving the above.

Suppose you're given a read-only array of nnn distinct items SSS in a fixed order, which you must preprocess into a data structure TTT in preparation for a sequence of NNN random index queries, i.e. queries which request the position of any sss in SSS.

Assume N>>nN >> nN>>n is extremely large, potentially infinite.

The map of each item sis_isi​ to its index iii should use only a constant number of simple arithmetic operations from the set {+,-,>>,<<,^}.

One can come up with a number of scenarios where this problem could arise in real examples. For example, when N=∞N = \inftyN=∞, this is like asking to construct the ultimate read-only cache for a streaming algorithm under the assumption of randomized (uniform) access.

First, let's acknowledge the elephant in the room. Isn't this extremely simple? Why don't we bust out the secret weapon: HashMap-uh. Those have O(1)O(1)O(1) lookup complexity, right?

Let's try in Python. I sampled 10k distinct random integers from a 100k range. Then, I mapped each integer to its index via a dictionary comprension.

import numpy as np 
U = range(1e6)
S = np.random.choice(U, size=1e5, replace=False)
h = { s : i for i,s in enumerate(S) }

Sure enough, this creates a valid mapping. And it's pretty fast.

all([h[s] == i for i, s in enumerate(S)]) 
## True

timeit.timeit(lambda: [h[s] for s in S], number=100)/100
## 0.002030719479999945

Hashing all 10k elements takes about ~2 ms. Not bad. But does this solve the problem? Is the O(1)O(1)O(1) lookup guaranteed?

Spoiler alert: It is not. The time complexity given at python.org states the complexity of accessing elements in a dictionary is average case O(1)O(1)O(1) and worst case O(n)O(n)O(n) (amortized). Because integers are such simple objects, Python's default hashing technique does quite well. More details on Pythons dict implementation can be found here SE post and this blog post. The actual dict.c source is here.

Experimenting with dict

Let's get an idea of how good Python is at hashing. Suppose we create a dumb wrapper class for Python's standard int type (which, incidentally, is actually a bignum type?)

@dataclass
class Int():
  val: int = 0

The dataclass decorator just adds __repr__ and __init__ methods. Now, let's replace our set S with these wrapped types

S = [Int(i) for i in S]

Also, let's rebuild our hash map h and benchmark the average access time for 10k elements.

h = { s : i for i,s in enumerate(S) }
timeit.timeit(lambda: [h[s] for s in S], number=100)/100
## 0.0020396138799969776

Still about ~2 ms, nothing has really changed. Now, consider the following code:

Int.__hash__ = lambda self: 0
h = { s : i for i,s in enumerate(S) }
timeit.timeit(lambda: [h[s] for s in S], number=100)/100
## 11.08112517602

Holy crap. That is ≈5433 x\approx 5433 \, \text{x}≈5433x slower than before. What in the world just happened?

The __hash__ dunder is a magic method that Python relies on to build dictionaries. By returning a constant---effectively the worst hash function we could use---we've essentially reduced dict's access time to O(n)O(n)O(n). Re-running the example with Int.__hash__ = lambda self: hash(self.val) restores the ~2 ms runtime.

We would've been better off sorting S into an array and relying on the O(nlog⁡n)O(n \log n)O(nlogn) lookup via something like bisect or np.searchsorted.

Assuming Python takes the hash of an element and modulo's it with the table size, we can inspect the average number collisions via:

len(S)/len(np.unique([hash(s)%len(S) for s in S]))
## ~ 1.578

So Python does quite well with integers, and in this sense the problem as given is effectively solved. However, the performance will inevitably depends on the size of U\mathcal{U}U in relation to the size of SSS. For example, if we have ∣S∣=12∣U∣\lvert S \rvert = \frac{1}{2}\lvert \mathcal{U} \rvert∣S∣=21​∣U∣, performance drops an order of magnitude:

U = range(2*100000)
S = np.random.choice(U, size=100000, replace=False)
h = { s : i for i,s in enumerate(S) }
timeit.timeit(lambda: [h[s] for s in S], number=100)/100
# 0.021988959899990734

Moreover, if the set SSS is comprised of not just simple integers but some composite type, the performance could be drastically different.

The problem I was concerned with was similar to the application example. I have a large set SSS of integer-like types that are fixed, and I would like to have a fantastistically fast way of mapping those integers to their indices.

Math

Let's get some definitions out of the way. Let U\mathcal{U}U denote a universe of hashable elements and let S⊆US \subseteq \mathcal{U}S⊆U denote any subset of U\mathcal{U}U with which we want create a hash table. A hash function h:S→[m]h: S \to [m]h:S→[m] is a mapping between a set SSS and table T[1:m]T[1:m]T[1:m] of size m=∣T∣m = \lvert T \rvertm=∣T∣. In practice, it is often the case that h:S→Zh : S \to \mathbb{Z}h:S→Z and hhh is restricted to [m][m][m] via a modulo operation. A hash function is said to be perfect with respect to SSS if there are no collisions:

∀s,s′∈S,h(s)≠h(s′)  ⟹  s≠s′\forall s,s' \in S, \quad h(s) \neq h(s') \implies s \neq s' ∀s,s′∈S,h(s)=h(s′)⟹s=s′

In math speak, one notices that the definition of a perfect hash function is essentially that of an injective function. If m=nm = nm=n, i.e. the size of hash table matches the size of input set SSS, then the definition coincides with a bijection. Bijections, by definition, are invertible maps. Thus, the problem can be re-stated as finding a invertible map h:S→[n]h: S \to [n]h:S→[n] which uses O(1)O(1)O(1) operations to evaluate.

In general, hash functions are imperfect: it is entirely possible to map two distinct object s≠s′s \neq s's=s′ to the same hash value h(s)=h(s′)h(s) = h(s')h(s)=h(s′). If this happens, we say sss collides with s′s's′ in hhh. In order to recover the correct mapping, one typically resolves to some sort of collision resolution strategy, such as quadratic hashing, open addressing, linear probing, etc.
